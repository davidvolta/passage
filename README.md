# Passage
### Search and AI conversation across a book library

---

## What This Is

A web app for searching passages across a large book library and having AI conversations grounded in that content. Built for two people writing a book together.

---

## Pages

| Page | What it does |
|------|-------------|
| **Home** | Word particle animation drawn from the Notion workspace |
| **Passages** | Semantic search across the book library |
| **Channel** | AI conversations in Osho's voice, RAG-backed, three modes |
| **Favorites** | Saved passages, persisted across sessions |

## Setup

```bash
pip install -r requirements.txt
```

`.env` needs:
```
OPENAI_API_KEY=...
QDRANT_URL=...
NOTION_TOKEN=...
NOTION_PAGE_ID=...
```

---

## Indexing Books

Add books to `books/pdf/` or `books/epub/`, then:

```bash
python scripts/convert.py   # PDF/EPUB → markdown
python scripts/ingest.py    # markdown → Qdrant embeddings
```

Idempotent — safe to re-run, only processes new files. First run on a large library costs roughly $1 in OpenAI API usage.

---

## Running the App

```bash
uvicorn app.main:app --reload
```

Open `http://localhost:8000`. Qdrant must be reachable at `QDRANT_URL`.

## Running Tests

```bash
pytest tests/ -v
```

## Notion Sync

The home page animation pulls words from a Notion workspace. Set `NOTION_TOKEN` and `NOTION_PAGE_ID` in your environment, then run manually or let the daily cron handle it:

```bash
python scripts/update_notion_words.py
```

In production this runs automatically at midnight UTC via Railway's cron scheduler (`railway.toml`). The output file (`words_notion.json`) lives on the Railway `/data` volume and persists across deploys.

---

## Deploying

```bash
git push  # Railway auto-deploys on push to master
```

The `/data` volume persists `saved_passages.json` and `words_notion.json` across deploys. The daily cron job is defined in `railway.toml`.

## Key Design Decisions

**Why Markdown as intermediate format?**
Every book becomes a human-readable `.md` file before chunking. If a passage looks wrong in search results, you can open the file and see exactly what the converter produced.

**Why Qdrant remote-only?**
Originally ran in local mode (a folder). Now runs as a remote server — same API, no schema changes, shared across deploys.

**Why OpenAI embeddings?**
One-time cost (~$1 for a large library), no local GPU or model management, and `text-embedding-3-small` is genuinely better for semantic retrieval than local alternatives at this scale.

**Why 200-word chunks?**
Smaller chunks produce more precise semantic matches. Results feel targeted rather than overwhelming.
